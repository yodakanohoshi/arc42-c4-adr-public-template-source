# 9. Architecture Decisions

arc42本文にはADRの索引と現在の状態を記載し、判断の詳細は個別ADRへ分離する。

## 9.1 ADR一覧

[ADR一覧ページ](adr/index.md)を参照する。

## 9.2 ADR作成基準

次のいずれかに該当する判断をADRとして残す。

- 変更コストが高い
- 複数チーム・システムへ影響する
- セキュリティ、信頼性、運用へ大きく影響する
- 複数案を比較して決めた
- 後から判断理由を知る必要がある
- 契約、責任分界、データ所有へ影響する

## 9.3 運用ルール

- 承認済みADRの結論を上書きしない
- 判断を変更するときは新しいADRで置換する
- C4要素、要件ID、品質要件IDを相互参照する
- 小さな実装詳細や一時的な作業はADRにしない
- 未決の提案はDesign Doc / RFCでレビューし、決定後にADR化する
